WMS 지식베이스 — 개요 및 운영 가이드

소스코드 역공학 → 업무지식 문서화 → 커스터마이징 소스 생성 전체 프로세스 정리

1. 목적 및 배경

WMS 패키지는 이미 F/W 위에 입고·출고·재고·기준정보 업무가 구현된 소스로 고객에게 제공된다. 매번 신규 구축하면 비용이 크기 때문에 커스터마이징 요소만 변경하여 소스를 직접 재사용하는 방식으로 운영한다.

Claude Code를 활용하여 커스터마이징 효율을 높이기 위해 메뉴코드 단위로 7개의 업무지식 문서를 생성하고, 이를 기반으로 소스를 생성·수정한다.
2. WMS 패키지 레이어 구조
F/W (Framework) 공통 기능·공통 컴포넌트 · 인증·권한·로깅 등 플랫폼 기반
WMS Package 입고 · 출고 · 재고 · 기준정보 업무 소스 (메뉴코드 단위 관리)
Customer Customizing 고객사별 요구사항에 맞춘 변경 · 추가 소스 (메뉴코드 순번으로 분기)
커스터마이징은 F/W를 건드리지 않고 WMS 패키지 소스를 기반으로 진행한다. 변경 범위에 따라 기존 메뉴코드를 수정하거나 새 순번을 추가한다.
3. 전체 작업 사이클
1

역공학

기존 소스(BE Java + FE Vue)를 읽어 7개 문서 자동 생성

2

문서 수정

커스터마이징 요건에 맞게 7개 문서를 수정

3

소스 생성

Claude Code가 7개 문서를 기반으로 BE/FE 소스 생성

4

보완 프롬프트

부족한 부분은 추가 프롬프트로 지시하여 소스 보완

5

문서 업데이트

개발 완료 후 소스를 재분석하여 7개 문서를 최신화

⚠️ 보완 시 추가된 내용이 7개 문서에 반드시 반영되어야 한다. 문서와 소스의 동기화가 핵심.

5. 메뉴코드 순번 전략

메뉴코드 끝 두 자리 순번(01, 02, ...)으로 변형 버전을 관리한다.

📌 기준 메뉴 (예: MDPD01)

  • WMS 패키지 기본 품목 메뉴
  • 표준 UI 레이아웃 + 표준 BE 로직
  • 신규 고객 구축의 출발점

🔀 변형 메뉴 (예: MDPD02)

  • 기준 메뉴를 바탕으로 커스터마이징한 버전
  • 소스는 별도 유지, 메뉴관리에서 코드만 교체
  • 추후 다른 고객에게 재사용 가능

순번을 증가시키는 기준 (둘 중 하나라도 해당되면 신규 순번 추가)

변경 유형 예시 처리 방법
UI Layout 변경 단일 그리드 → 상하 분할 그리드, 팝업 구조 변경 신규 순번 추가 (기존 소스 유지)
BE 테이블 추가 연동 테이블이 추가되어 JOIN 구조 변경 신규 순번 추가 (기존 소스 유지)
BE 플로우 변경 처리 단계가 달라지거나 트랜잭션 경계 변경 신규 순번 추가 (기존 소스 유지)
컬럼 추가만 기존 그리드에 컬럼 몇 개 추가, 조건 추가 기존 메뉴코드 수정 (순번 유지)
핵심 목표: 메뉴 코드 라이브러리 누적 → 메뉴관리 프로그램에서 코드만 변경 → 개발 없이 또는 최소 커스터마이징으로 신규 고객 대응 가능
MDPD01 ─── 기본 품목 (패키지 표준)
    └── 고객 A: MDPD01 그대로 사용
    └── 고객 B: 컬럼 추가 → MDPD01 수정

MDPD02 ─── 레이아웃 변형 (상하 분할)
    └── 고객 C: MDPD02 그대로 사용 ← 개발 없음
    └── 고객 D: 소폭 커스터마이징 → MDPD02 수정

MDPD03 ─── 추가 테이블 포함 버전
    └── 고객 E: MDPD03 적용 예정 …
