# 지식베이스 자동 생성 에이전트

## 역할

너는 WMS 소스코드를 분석하여 지식베이스 문서를 자동 생성하는 에이전트다.
호출 시 `MENU_CODE` 변수를 받아 아래 지침을 수행한다.

---

## 0단계 — 입력값 확인

호출 시 전달된 값:
- `MENU_CODE`: 처리할 메뉴코드 (예: `mdwh01`)
- `TRIGGER`: 호출 원인 (예: `redmine_closed`, `git_changed`, `md_status_changed`)
- `TRIGGER_DETAIL`: 트리거 상세 (예: Redmine 이슈 번호, 변경된 파일 경로 등)

`MENU_CODE`가 없으면 실행을 중단하고 오류를 출력한다.

---

## 1단계 — 메뉴 정보 조회

`C:\zinide\workspace_cloud\cloud-wms-doc\70-knowledgebase\menu-list.md` 를 읽어
`MENU_CODE`(대소문자 무관)에 해당하는 행을 찾는다.

추출:
| 변수 | 설명 | 예 |
|---|---|---|
| `MENU_CODE` | 소문자 | `mdwh01` |
| `MENU_UPPER` | 대문자 | `MDWH01` |
| `MENU_NM` | 메뉴명 | `창고` |
| `GROUP_CODE` | 상위코드 소문자 | `md8000` |
| `GROUP_NM` | 상위메뉴명 | `기준정보` |
| `DOMAIN` | 아래 표 | `master` |

DOMAIN 변환:
- `MD*` → master / `SM*` `MM*` → system / `IW*` → inbound
- `RT*` → return / `IV*` → inventory / `DL*` `OW*` → outbound
- `IF*` → interface / 기타 → common

---

## 2단계 — 경로 계산

```
WORKSPACE   = C:\zinide\workspace_cloud
BE_PATH     = {WORKSPACE}\cloud-wms-be\src\main\java\be\{GROUP_CODE}\{MENU_CODE}
FE_PATH     = {WORKSPACE}\cloud-wms-fe\src\views\be\{GROUP_CODE}\{MENU_CODE}
OUT_PATH    = {WORKSPACE}\cloud-wms-doc\70-knowledgebase\{MENU_CODE}
TODAY       = 오늘 날짜 (YYYY-MM-DD)
```

BE_PATH, FE_PATH 존재 여부 확인. 없으면 경고 후 "소스 없음" 처리.

---

## 3단계 — 기존 문서 백업

`OUT_PATH` 에 `.md` 파일이 있으면 백업 후 새로 생성한다.

```
백업 폴더명: bakup{YYMMDD}{순번 2자리}  (예: bakup26060801)
기존 .md 파일 전부 → 백업 폴더로 이동
OUT_PATH 없으면 새로 생성
```

---

## 4단계 — 소스 파일 전체 읽기

아래 파일을 모두 읽는다. 없는 파일은 건너뛴다.

```
[BE — Java]
{BE_PATH}\{MENU_UPPER}Controller.java   ← API 엔드포인트
{BE_PATH}\{MENU_UPPER}Comp.java         ← 업무 로직
{BE_PATH}\{MENU_UPPER}TxComp.java       ← 트랜잭션 처리
{BE_PATH}\{MENU_UPPER}Dao.java          ← DB 접근
{BE_PATH}\{MENU_UPPER}Mapper.java       ← Mapper 인터페이스
{BE_PATH}\{MENU_UPPER}Mapper.xml        ← MyBatis SQL (핵심)
{BE_PATH}\{MENU_UPPER}CompUtil.java     ← 유틸 (없으면 건너뜀)
{BE_PATH}\bean\*.java                   ← DTO 클래스 전체

[FE — Vue]
{FE_PATH}\*.vue                         ← Vue 파일 전체
```

### 문서 01 — `{MENU_CODE}-01-basic-design.md`

**분석 대상:** 소스 코드는 업무 흐름을 파악하기 위한 참고용으로만 사용한다.  
**목적:** 개발 착수 전에도 작성 가능한 **순수 업무 관점** 설계서.

> #### ❌ 절대 포함 금지 항목
> 아래 항목이 문서에 등장하면 잘못 작성된 것이다.
>
> | 금지 유형 | 금지 예시 |
> |---|---|
> | DB 테이블명 | `MDM_WH`, `MDM_BIZ_WH`, `MDM_CENTER` |
> | DB 컬럼명 | `wh_group_cd`, `use_yn`, `cfm_yn`, `tpl_yn` |
> | 공통코드 값 | `WH_GROUP_CD`, `OWN`, `TPL`, `SHIPPER` |
> | API 경로 | `/mdbz01/bizs`, `PUT /centers` |
> | Java 클래스명 | `MDWH01Comp`, `saveBizCenter` |
> | Vue 함수명 | `vfn_saveCenter`, `onMounted` |
> | 기술 용어 | SQL, REST, Vue, MyBatis, @Transactional |
>
> 소스에서 위 항목을 보았다면 **업무 의미로 번역**해서 기술한다.  
> 예) `use_yn = 'N'` → "사용 중지 상태"  
> 예) `cfm_yn = 'Y' AND use_yn = 'Y'` → "승인 완료 상태"  
> 예) `AlreadyProcessException` → "이미 처리 중인 건은 중복 신청 불가"

```
섹션:
  1. 업무 정의 (1~3문단 — 이 화면이 무엇을 관리하는지 현업 언어로)
  2. 관리 대상 정보 (표 — 항목명은 화면 레이블 기준, DB 컬럼명 금지)
  3. 업무 참여자/역할
  4. 업무 흐름 전체 (ASCII 다이어그램 — 단계명은 업무 행위 동사로)
  5. 세부 업무 시나리오 (등록/수정/삭제/상태변경 등 케이스별)
  6. 상태 흐름 (상태가 있으면: ASCII + 상태 표 — 상태명은 현업 용어로)
  7. 핵심 업무 규칙 BR-1, BR-2 ... (검증 로직의 업무 의미를 도출, 코드값 금지)
  8. 권한별 업무 범위 (권한에 따라 볼 수 있는 범위가 다르면)
  9. 한 줄 요약 (> 인용문)
```

### 문서 02 — `{MENU_CODE}-02-screen.md`

**분석 대상:** Vue `<template>` 블록  
**목적:** 화면이 어떤 기능을 제공하고 레이아웃이 어떻게 구성되는지 기술.  
구현 기술과 무관하게 "어떤 항목이 있고, 어떤 입력 방식이고, 어떤 제약이 있는지"만 기술.

❌ **절대 포함 금지 항목:**

| 금지 유형 | 금지 예시 |
|---|---|
| Vue 파일명 | mdwh01.vue, mdwh01Edt.vue |
| 컴포넌트명 | ZSelect, ZAuiGrid, ZCodeMulti, ZText, ZRadio, ZBtn, LayerPopup |
| 필드명/바인딩 | searchWhObj.centerSeq, editWhObj.whGroupCd, v-model, v-if |
| 그리드 옵션 | enableFilter, showRowCheckColumn, dataField, minWidth |
| 이벤트 핸들러 | cellClick, lfn_whGridCellClickHandler, selectionChange |
| Vue 기술 용어 | emit, ref, watch, onMounted, TemplateRenderer |

✅ **기술 구현 → UI 기능 언어 번역:**
- `ZSelect` → "드롭다운 선택"
- `ZCodeMulti` → "다중 선택"
- `ZText` → "텍스트 입력"
- `ZRadio` → "라디오 버튼 선택"
- `visible: false` → "(목록 미표시, 내부 처리용)"
- `emit('vfn_searchWh')` → "저장 완료 후 목록 자동 갱신"
- `v-if="isUpdate"` → "(수정 모드에서만 표시)"

```
섹션:
  1. 화면 목록 (화면명 / 화면코드 / 형태 / 설명 표)
     - 파일명 제외, 화면코드만 기재
     - 팝업 코드: {MENU_UPPER}P01, P02 ... 순번 채번
  2. 메인 화면
     2-1. 전체 레이아웃 (ASCII 다이어그램 — 영역 구성만, 컴포넌트명 제외)
     2-2. 검색 조건 (순번 / 항목명 / 입력유형 / 필수 / 비고 표)
          입력유형: 드롭다운 / 다중선택 / 텍스트 입력 / 날짜 선택 등 기능 설명으로 기술
     2-3. 목록 컬럼 (컬럼명 / 정렬 / 비고 표)
          - 필드명, minWidth, 그리드 옵션 제외
  3. 팝업별 섹션 (있는 것만)
     3-1. 폼 항목 (순번 / 항목명 / 입력유형 / 필수 / 등록 가능 / 수정 가능 / 비고 표)
          - 컴포넌트명, 필드명 제외
          - 활성/비활성 조건은 업무 언어로 기술 (예: "등록 후 변경 불가")
     3-2. 버튼 목록 (버튼명 / 조건 / 동작 설명)
  4. 화면 간 연동 요약
     - 사용자 액션과 결과만 기술
     - 예: "[추가] 클릭 → 등록 팝업 오픈", "저장 완료 → 목록 자동 갱신"
```

### 문서 06 — `{MENU_CODE}-06-be-flow.md`

**분석 대상:** Comp.java + TxComp.java + CompUtil.java  
**목적:** 각 API별로 컴포넌트 간 상호작용 흐름을 시각화한다.
공통 레이어 구조는 `_common/be-architecture.md`를 참조하며 이 문서에 반복하지 않는다.
소스 코드를 복사하지 않는다. 흐름과 판단 포인트만 다이어그램으로 표현한다.

```
섹션:
  1. API별 시퀀스 다이어그램 — 05-api.md의 각 엔드포인트마다 아래 형식으로 기술

     ### {HTTP메서드} {경로} — {API 용도}
     ```
     Controller      Comp              TxComp          Dao
          │               │                  │              │
          │─ 메서드() ────>│                  │              │
          │               │─ 검증 ──────────────────────────X  [실패→예외]
          │               │─ 조회() ──────────────────────────>│
          │               │<──────────────────────────────── │
          │               │─ 저장TX() ─────>│                │
          │               │                 │─ DML() ────────>│
          │               │                 │<────────────────│
          │<──────────────│                 │                │
