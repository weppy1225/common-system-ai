# BE/FE 문서 구조와 FE 자동화 전략

## 1. 목적

이 문서는 백엔드 저장소와 프론트 저장소의 문서 책임을 분리하고, 프론트 자동화를 어떤 방식으로 설계하는 것이 적절한지 정리한 운영 전략 문서다.

핵심 원칙은 다음과 같다.

- 백엔드 저장소는 백엔드 규칙과 FE 입력 산출물을 책임진다.
- 프론트 자동화는 프론트 저장소에서 운영한다.
- 화면 설계, UI 구현, API 연동, E2E 테스트는 FE 저장소 기준으로 분리한다.

---

## 2. 현재 백엔드 저장소 문서 구조

현재 백엔드 저장소의 운영 기준 구조는 아래와 같다.

```text
cloud-wms-be/
├── CLAUDE.md
├── .claude/
│   ├── commands/
│   ├── rules/
│   └── skills/
└── DEV_DOC/ai-docs/
    ├── 00-memory/
    ├── 10-database/
    ├── 20-backend/
    └── 30-interface/
```

역할 분리는 아래처럼 본다.

- `CLAUDE.md`: 최상위 프로젝트 규칙
- `.claude/commands`: 작업 절차
- `.claude/rules`: 판단 기준
- `DEV_DOC/ai-docs/10-database`: DB 지식과 명세
- `DEV_DOC/ai-docs/20-backend`: 백엔드 규칙, 가이드, spec, API
- `DEV_DOC/ai-docs/30-interface`: 인터페이스 규칙, 가이드, spec, API

---

## 3. 권장 백엔드 문서 구조

백엔드 저장소는 현재 구조를 유지하되, 역할이 겹치지 않도록 아래 원칙으로 운영한다.

```text
DEV_DOC/ai-docs/
├── 00-memory/      ← 프로젝트 공통 메모리, 용어, 운영 전략
├── 10-database/    ← 테이블, 코드값, DB 규칙, DB 변경 이력
├── 20-backend/     ← 백엔드 규칙, 가이드, spec, API 산출물
└── 30-interface/   ← IF 규칙, 가이드, spec, API 산출물
```

백엔드 저장소 안에는 프론트 구현 규칙을 억지로 넣지 않는 것이 좋다.

이유:

- 레포 책임이 섞인다
- FE 규칙 변경이 BE 저장소 변경으로 이어진다
- 화면 설계와 UI 구현 규칙은 FE 코드베이스와 함께 움직이는 편이 안전하다

---

## 4. 프론트 자동화는 왜 별도 저장소에서 해야 하는가

프론트 자동화는 다음 이유로 `cloud-wms-fe`에서 운영하는 것이 적절하다.

- 실제 컴포넌트 구조, 라우팅, 상태관리, 공통 UI 규칙이 FE 저장소에 있다
- 화면 구현 규칙은 백엔드보다 프론트 코드 컨텍스트에 더 강하게 의존한다
- API 연동, 화면 상태, 테스트 도구는 FE 저장소와 함께 변한다
- 백엔드 저장소에서 프론트 구현까지 지시하면 문서 충돌과 책임 경계 혼선이 생긴다

따라서 백엔드 저장소는 프론트 입력 산출물을 생산하는 역할로 제한한다.

---

## 5. 권장 프론트 문서 구조

프론트 저장소에는 아래 구조를 권장한다.

```text
cloud-wms-fe/
├── CLAUDE.md
├── .claude/
│   ├── commands/
│   │   ├── 01-plan/
│   │   │   ├── fe-spec.md
│   │   │   └── fe-task.md
│   │   ├── 02-dev/
│   │   │   ├── fe-dev-screen.md
│   │   │   ├── fe-dev-api.md
│   │   │   └── fe-dev-all.md
│   │   └── 03-support/
│   │       └── fe-output.md
│   ├── rules/
│   └── skills/
└── DEV_DOC/ai-docs/
    └── 30-frontend/
        ├── 00-frontend-overview.md
        ├── 10-architecture/
        ├── 20-rule/
        ├── 30-convention/
        ├── 40-guide/
        ├── 80-spec/
        └── 90-ui/
```

권장 역할:

- `20-rule`: 라우트, 파일명, 네이밍, 권한 규칙
- `30-convention`: 컴포넌트, 상태관리, API 모듈, 타입 규칙
- `40-guide`: 화면/팝업/그리드 작성 가이드
- `80-spec`: 화면별 명세와 task
- `90-ui`: 구현된 화면 기준 문서와 산출물

---

## 6. 권장 FE 자동화 워크플로우

프론트 자동화는 아래 순서를 권장한다.

### 6.1 fe-spec

목적:

- 화면 요구사항 정리
- 그리드, 폼, 팝업, 버튼 액션 정의
- API 매핑 테이블 작성

주요 산출물:

- 화면 명세서
- 상태 정의
- API 연동 목록

### 6.2 fe-task

목적:

- 구현 단위를 쪼개어 누락 없이 작업하기 위함

작업 항목 예시:

- 라우트 등록
- 메인 화면 컴포넌트
- 팝업 컴포넌트
- API 모듈
- 타입 정의
- 권한 처리
- 테스트 항목

### 6.3 fe-dev-screen

목적:

- 화면 컴포넌트와 레이아웃 생성
- 더미 데이터 기준 렌더링 확인

### 6.4 fe-dev-api

목적:

- API 모듈, 타입, 요청/응답 매핑 구현
- 로딩, 에러, 빈 데이터 처리 연결

### 6.5 fe-dev-all

목적:

- 구조가 명확한 화면에 대해 화면 + API + 기본 상태 처리를 한 번에 생성

사용 조건:

- 레퍼런스 화면이 명확한 경우
- 상태 전이가 단순한 경우
- 팝업, 탭, 복합 편집 흐름이 복잡하지 않은 경우

### 6.6 fe-output

목적:

- FE 구현 결과를 QA 와 운영 인수용으로 문서화

산출물 예시:

- 라우트 정보
- 화면 경로
- 사용 API 목록
- 필수 사용자 액션
- 테스트 포인트

---

## 7. 백엔드에서 프론트로 넘겨야 할 산출물

백엔드 저장소는 아래 산출물을 FE 입력으로 제공하는 것을 기본으로 한다.

- `spec.md`
- 화면 요구사항 문서 또는 기능 설명 문서
- `{YYYYMMDD}_output.md`
- 샘플 request / response JSON

최소 기준:

- API URL
- HTTP Method
- 요청 필드
- 응답 필드
- 경고/에러 처리 규칙
- 인증 또는 권한 정보

---

## 8. FE 문서에 반드시 들어가야 할 규칙

프론트 저장소 문서에는 최소한 아래 항목이 있어야 한다.

- 라우트 규칙
- 파일명 규칙
- 화면 / 팝업 / 그리드 네이밍 규칙
- API 모듈 위치 규칙
- 타입 정의 위치 규칙
- 공통 컴포넌트 재사용 규칙
- 권한별 버튼 표시 규칙
- 로딩 / 에러 / 빈 데이터 처리 규칙
- 테스트 기준

테스트 기준은 아래 수준까지 명시하는 것이 좋다.

- 렌더링 확인
- API 모킹 기준
- 사용자 액션 처리
- 권한별 분기
- 빈 데이터 / 에러 응답 처리

---

## 9. 운영 기본 원칙

1. 백엔드 저장소는 FE 입력 산출물까지 책임진다.
2. FE 구현 규칙과 자동화 명령은 FE 저장소에서 운영한다.
3. 백엔드 저장소 안에 FE 구현 절차를 과도하게 넣지 않는다.
4. 화면 설계, UI 구현, API 연동, E2E 테스트는 FE 코드베이스 기준으로 관리한다.
