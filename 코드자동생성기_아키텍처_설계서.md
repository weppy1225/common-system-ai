## 1. 개요

### 1-1. 목적

이 문서는 cloud-wms 프로젝트에서 Claude Code가 **DB → BE → FE** 순서로 코드를 자동 생성하는 전체 흐름과, 각 단계를 연결하는 **프롬프트 체인 설계**를 정의한다.

### 1-2. 코드 자동 생성기란

> 사람이 "메뉴코드"와 "요건"을 입력하면, Claude Code가 DB 스키마 → 백엔드 레이어 → 프론트엔드 화면까지 자동으로 생성하는 AI 기반 코드 생성 파이프라인이다.

### 1-3. 설계 원칙

| 원칙 | 설명 |
|---|---|
| **DB 우선** | DB 스키마가 확정된 후 BE 코드를 생성하고, BE 스펙이 확정된 후 FE 코드를 생성한다 |
| **단방향 흐름** | DB → BE → FE 방향으로만 흐른다. FE가 DB를 직접 참조하지 않는다 |
| **스펙 문서가 계약** | BE와 FE 사이의 유일한 계약은 `80-spec/{메뉴코드}/spec.md` 이다 |
| **레이어별 검증** | 각 레이어 생성 후 JUnit/ESLint 검증을 통과해야 다음 레이어로 진행한다 |
| **KB 의존** | 코드 생성은 KB(rules/, skills/, ai-docs/)에 정의된 규칙을 반드시 따른다 |

## 2. 전체 생성 흐름

### 2-1. 3-Layer 생성 파이프라인

```
사용자 입력
(메뉴코드 + 요건)
       │
       ▼
┌──────────────────────────────────────────────────────┐
│  PHASE 0: 사전 분석                                   │
│  기존 자산 탐색 + 도메인 컨텍스트 로드                  │
└──────────────────────────────────────────────────────┘
       │
       ▼
┌──────────────────────────────────────────────────────┐
│  PHASE 1: DB 생성                                     │
│  테이블 설계 → DDL 생성 → DB 적용 → 스키마 문서화       │
└──────────────────────────────────────────────────────┘
       │  spec.md (DB 스키마 포함)
       ▼
┌──────────────────────────────────────────────────────┐
│  PHASE 2: BE 생성                                     │
│  명세 → 계획 → Mapper → Dao → Comp → Controller       │
└──────────────────────────────────────────────────────┘
       │  spec.md (API 명세 포함)
       ▼
┌──────────────────────────────────────────────────────┐
│  PHASE 3: FE 생성                                     │
│  BE 스펙 파싱 → Vue 파일 생성 → 메뉴 등록 → 검증       │
└──────────────────────────────────────────────────────┘
       │
       ▼
    완료 (산출물 정리)
```

### 2-2. 레이어 간 인터페이스

```
DB Layer ──────────────────────────────────────────────
  │  테이블명, 컬럼, PK/FK, 공통코드
  │  전달 매체: DEV_DOC/ai-docs/10-database/90-schema/20-tables/{테이블명}.md
  ▼
BE Layer ──────────────────────────────────────────────
  │  API URL, HTTP 메서드, Request/Response 필드
  │  전달 매체: DEV_DOC/ai-docs/20-backend/80-spec/{메뉴코드}/spec.md
  ▼
FE Layer ──────────────────────────────────────────────
     Vue 컴포넌트, API 호출 코드, 화면 레이아웃
```

## 3. DB 생성 흐름

### 3-1. DB 생성 단계

```
[입력]
요건 설명 (자연어)
기존 관련 테이블명

    │
    ▼
┌─────────────────────────────────────────────┐
│ STEP 1. 기존 스키마 분석                      │
│ @db-doc-reader {관련 테이블명}                │
│ → 기존 테이블 구조, 컬럼, PK/FK, 공통코드 파악  │
└─────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────┐
│ STEP 2. 신규 테이블/컬럼 설계                  │
│ /design-db                                   │
│ → 정규화, 컬럼 타입, NOT NULL, 기본값 결정     │
│ → 네이밍 규칙 적용 (db-convention.md 참조)    │
│ → 소프트 삭제 컬럼 (use_yn / del_yn) 추가     │
└─────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────┐
│ STEP 3. DDL 스크립트 생성                     │
│ → CREATE TABLE / ALTER TABLE / COMMENT       │
│ → 인덱스 설계                                 │
│ → 공통코드(_cd) 컬럼 정의                     │
└─────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────┐
│ STEP 4. DB 적용                              │
│ /util-db-apply                               │
│ → Docker PostgreSQL 컨테이너에 DDL 실행       │
│ → 적용 결과 확인                              │
└─────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────┐
│ STEP 5. 스키마 문서화                         │
│ → DEV_DOC/ai-docs/10-database/90-schema/    │
│    20-tables/{테이블명}.md 자동 생성           │
│ → 공통코드 추가 시 01-common-code.md 갱신     │
└─────────────────────────────────────────────┘

[출력]
DDL 스크립트
테이블 스키마 문서 (.md)
```

### 4-3. BE 자동화 훅 (위반 감지)

코드 생성 중 아래 패턴이 감지되면 자동으로 경고하고 수정을 요구한다.

| 위반 패턴 | 감지 시점 | 조치 |
|---|---|---|
| TxComp 외 `@Transactional` 선언 | 파일 저장 후 | 경고 + `@Transactional` 제거 요구 |
| `Mapper.xml`에서 `DELETE FROM` | 파일 저장 후 | 경고 + `use_yn='N'` 소프트 삭제로 교체 |
| `ZTEST_*.java`에서 `@Test` 삭제 | 파일 저장 후 | 경고 + 테스트 복원 요구 |
| 커밋 전 민감정보 포함 | git commit 전 | 경고 + 커밋 차단 |

## 5. FE 생성 흐름

### 5-1. FE 생성 단계

```
[입력]
메뉴코드
BE spec.md (BE STEP 4 출력)

    │
    ▼
┌─────────────────────────────────────────────┐
│ STEP 1. BE 스펙 파싱                          │
│ @be-spec-reader {메뉴코드}                    │
│ → API 목록, Request/Response 필드 추출        │
│ → 복합키, 공통코드 컬럼 추출                   │
│ → 우선순위: {YYYYMMDD}_output.md > spec.md   │
└─────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────┐
│ STEP 2. Vue 파일 생성                         │
│ /new-menu-from-be-spec {메뉴코드}             │
│                                              │
│  2-1. 메뉴 문서 생성                          │
│       ai-docs/20-frontend/60-menus/          │
│       {업무군}/{메뉴코드}/menu.md             │
│                                              │
│  2-2. 리스트 화면 생성                         │
│       src/views/be/{업무군}/{메뉴}/{메뉴}.vue  │
│       (검색 + 그리드 + CRUD 패턴)             │
│                                              │
│  2-3. 등록/수정 화면 생성                      │
│       src/views/be/{업무군}/{메뉴}/{메뉴}Edt.vue│
│                                              │
│  2-4. 자기 검증 (5개 항목)                     │
│       HTTP 메서드 / URL / 복합키 / 응답네이밍 /  │
│       문서 일치도                              │
└─────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────┐
│ STEP 3. 라우터 등록                           │
│ → src/router/index.js 에 메뉴 경로 추가       │
│ → 좌측 메뉴 트리 연결                         │
└─────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────┐
│ STEP 4. 코드 검증                             │
│ @fe-code-reviewer                            │
│ → 함수 접두사 규칙 (vfn_/gfn_/sfn_/lfn_)    │
│ → import 순서 점검                           │
│ → 금지사항 검출 (axios 직접, swal 직접 등)    │
└─────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────┐
│ STEP 5. 계약 검증                             │
│ /util-verify-menu {메뉴코드}                  │
│ → HTTP 메서드 일치 확인                       │
│ → URL 경로 일치 확인                          │
│ → 응답 네이밍 일치 확인                       │
└─────────────────────────────────────────────┘
```

## 6. 프롬프트 체인 설계

### 6-1. 프롬프트 체인이란

> 단일 프롬프트로 해결하기 어려운 복잡한 작업을, **여러 개의 프롬프트를 순서대로 연결**하여 앞 단계 출력을 다음 단계의 입력으로 전달하는 방식이다.

### 6-2. 전체 프롬프트 체인 맵

```
사용자: "메뉴코드 {XXXX01} 개발해줘"
    │
    ▼
[Chain 0: 컨텍스트 로드 체인]
    Prompt 0-1: 메뉴코드로 기존 소스 탐색
                "code-layer-explorer에게 {메뉴코드}로 기존 레이어 파일을 탐색해줘"
    Prompt 0-2: 관련 테이블 문서 로드
                "db-doc-reader에게 {테이블명}의 스키마 문서를 요약해줘"
    출력: 기존 소스 목록 + 테이블 스키마 요약
    │
    ▼
[Chain 1: DB 설계 체인]
    Prompt 1-1: 기존 테이블 분석
                "현재 테이블 구조를 파악하고 신규 컬럼이 필요한지 판단해줘"
    Prompt 1-2: DDL 생성
                "db-convention.md 규칙에 따라 DDL 스크립트를 작성해줘"
    Prompt 1-3: 스키마 문서 작성
                "생성된 테이블의 스키마 문서를 DEV_DOC/ai-docs/10-database 경로에 작성해줘"
    출력: DDL 스크립트 + 스키마 .md 파일
    │
    ▼
[Chain 2: BE 명세 체인]
    Prompt 2-1: 기능 명세 작성
                "스키마 문서와 요건을 바탕으로 spec.md를 작성해줘"
                입력: Chain 1 출력 + ui.md 요건
    Prompt 2-2: 개발 계획 작성
                "spec.md를 읽고 레이어별 메서드 목록이 담긴 plan.md를 작성해줘"
                입력: Prompt 2-1 출력 (spec.md)
    Prompt 2-3: 작업 분해
                "plan.md를 기반으로 task.md를 작성해줘"
                입력: Prompt 2-2 출력 (plan.md)
    출력: spec.md + plan.md + task.md
    │
    ▼
[Chain 3: BE 코드 생성 체인]
    Prompt 3-1: Mapper.xml 생성
                "plan.md의 Mapper 항목과 db-convention SKILL을 참조해서 Mapper.xml을 작성해줘"
                입력: plan.md + 스키마 문서
    Prompt 3-2: Mapper.java 생성
                "Mapper.xml의 메서드 시그니처에 맞게 Mapper.java를 작성해줘"
                입력: Prompt 3-1 출력
    Prompt 3-3: Dao 생성
                "Mapper.java를 참조해서 dao-writing-rules.md 규칙에 따라 Dao를 작성해줘"
                입력: Prompt 3-2 출력
    Prompt 3-4: TxComp 생성
                "plan.md의 TxComp 항목과 biz-framework SKILL을 참조해서 TxComp를 작성해줘"
                입력: Prompt 3-3 출력 + InvenManager 필요 여부 판단
    Prompt 3-5: Comp 생성
                "TxComp를 참조해서 Comp를 작성해줘"
                입력: Prompt 3-4 출력
    Prompt 3-6: Controller 생성
                "spec.md의 API 목록과 Comp를 참조해서 Controller를 작성해줘"
                입력: Prompt 3-5 출력 + spec.md
    Prompt 3-7: JUnit 생성 및 실행
                "각 레이어 파일을 참조해서 ZTEST_*.java를 작성하고 Docker로 실행해줘"
                입력: 각 레이어 파일
    출력: 6개 레이어 Java 파일 + JUnit 테스트 통과 확인
    │
    ▼
[Chain 4: FE 코드 생성 체인]
    Prompt 4-1: BE 스펙 파싱
                "be-spec-reader에게 {메뉴코드}의 spec.md를 파싱해서 FE용 정보를 추출해줘"
                입력: spec.md (Chain 2 출력)
    Prompt 4-2: menu.md 생성
                "파싱된 API 정보로 60-menus/{메뉴코드}/menu.md를 작성해줘"
                입력: Prompt 4-1 출력
    Prompt 4-3: 리스트 Vue 파일 생성
                "CRUD 패턴과 Z* 컴포넌트 가이드를 참조해서 {메뉴}.vue를 작성해줘"
                입력: Prompt 4-2 출력 + 30-convention 가이드
    Prompt 4-4: 등록 Vue 파일 생성
                "{메뉴}.vue와 spec.md를 참조해서 {메뉴}Edt.vue를 작성해줘"
                입력: Prompt 4-3 출력
    Prompt 4-5: 코드 검증
                "fe-code-reviewer에게 생성된 Vue 파일을 리뷰해줘"
                입력: Prompt 4-3, 4-4 출력
    출력: menu.md + {메뉴}.vue + {메뉴}Edt.vue + 검증 통과 확인
```

### 6-3. 체인 간 데이터 흐름

```
Chain 0 출력
  ├─ 기존 레이어 파일 목록    ──→  Chain 3 참조 (중복 방지)
  └─ 테이블 스키마 요약       ──→  Chain 1, 2, 3 참조

Chain 1 출력 (DDL + 스키마 문서)
  └─ 스키마 .md 파일         ──→  Chain 2 입력 (명세 작성 기반)
                                  Chain 3 입력 (Mapper 작성 기반)

Chain 2 출력 (spec + plan + task)
  ├─ spec.md                 ──→  Chain 3 입력 (API 목록)
  │                               Chain 4 입력 (FE 스펙 파싱)
  └─ plan.md                 ──→  Chain 3 입력 (메서드 목록)

Chain 3 출력 (BE 코드)
  └─ spec.md 확정본           ──→  Chain 4 입력 (FE 생성 기반)

Chain 4 출력 (FE 코드)
  └─ menu.md                 ──→  /sync-be-spec 갱신 트리거
```

### 6-4. 에이전트 역할 분담

| 에이전트 | 호출 시점 | 역할 | 출력 |
|---|---|---|---|

## 7. 프롬프트 설계 원칙

### 7-1. 프롬프트 작성 규칙

| 원칙 | 설명 |
|---|---|
| **컨텍스트 명시** | 프롬프트 시작 시 "어떤 KB를 참조할지"를 명시한다 |
| **입력 명시** | 이전 단계 출력물의 파일 경로를 명시한다 |
| **출력 명시** | 생성할 파일의 경로와 형식을 명시한다 |
| **검증 포함** | 각 프롬프트 끝에 자기 검증 항목을 포함한다 |
| **단일 책임** | 하나의 프롬프트는 하나의 레이어/파일만 담당한다 |

### 7-2. 표준 프롬프트 템플릿

```
## 컨텍스트
- 메뉴코드: {메뉴코드}
- 참조 KB: {rules 파일} + {guide 파일}
- 입력: {이전 단계 출력 파일 경로}

## 작업
{생성할 내용 및 규칙 적용 지시}

## 출력
- 파일 경로: {저장 경로}
- 형식: {Java/Vue/MD/XML}

## 검증
- [ ] {검증 항목 1}
- [ ] {검증 항목 2}
```

### 7-3. 레이어별 핵심 프롬프트 포인트

| 레이어 | 핵심 지시 | 검증 항목 |
|---|---|---|
| **Mapper.xml** | `db-convention SKILL` 참조, `DELETE FROM` 금지, Fragment 재사용 | SQL 문법 오류 없음, 공통코드 FROM 절 JOIN 포함 |
| **Dao** | `search*` / `select*` 메서드명 구분, `@Param` 2개 이상 시 적용 | 메서드명 규칙 준수 |
| **TxComp** | `@Transactional` 선언, InvenManager 경유, DocNoGenerator 사용 | @Transactional 위치, InvenManager 직접 쿼리 없음 |
| **Comp** | `@Transactional` 선언 금지, TxComp 호출 | @Transactional 없음 |
| **Controller** | HTTP 메서드 규약 (POST=리스트, PUT=등록), `ResponseData` 반환 | URL 규약, 메서드 규약 |
| **{메뉴}.vue** | Z* 컴포넌트만 사용, `axios` 직접 호출 금지, `gfn_` 함수 사용 | import 순서, 함수 접두사, swal 직접 호출 없음 |
| **{메뉴}Edt.vue** | `regBizSeq` 하드코딩 금지, POST → PUT/PATCH 구분 | 등록/수정 API 분리 |

## 9. 제약 및 예외 처리

### 9-1. 생성 불가 조건

| 조건 | 이유 | 대응 |
|---|---|---|
| DB 스키마 문서 없이 BE 코드 생성 | 임의 추정으로 오류 발생 가능 | `/design-db` 또는 수동 문서 작성 선행 |
| BE spec.md 없이 FE 코드 생성 | API 계약 불명확 | `/design-spec` 선행 필수 |
| JUnit 실패 상태에서 다음 레이어 진행 | 누적 오류 발생 | 실패 레이어 수정 후 재실행 |
| 운영 중 테이블에 컬럼 DROP | 데이터 손실 위험 | 관리자 승인 후 수동 처리 |

### 9-2. 부분 생성 (선택적 레이어)

SIF(외부연동) 개발처럼 표준 흐름과 다른 경우 별도 체인을 사용한다.

```
외부연동 개발 흐름:
/design-if-plan → /design-if-spec → /design-if-task → /dev-if-all
```
