# 30-src-index — 소스코드 위치 색인

"어느 메뉴/기능의 코드가 **어디 있나**"를 빠르게 찾기 위한 **색인**이다. 코드 요약본이 아니다 — AS-IS 정본은 소스 코드 자체.

## 무엇을 넣나
- 메뉴코드 → BE/FE 소스 경로 매핑 (예: `IWRQ01 → be/iw1000/iwrq01/`, FE `views/be/iw1000/iwrq01/`, 모바일 `views/bm/iw1000m/iwrq01m/`)
- 공통 모듈·유틸·인터페이스(SIF) 위치
- 그룹코드 ↔ 업무군 매핑

## 무엇을 넣지 않나
- 코드 내용 요약/설명 → 두지 않음 (필요하면 소스를 직접 읽는다)
- 설계 의도 → `spec/{프로젝트}/{메뉴}/`

## 규칙
- 경로 규약은 `.claude/rules/repo-paths.md`를 따른다.
- 도출 규칙: 메뉴코드 소문자 → `be/{그룹}/{메뉴}`, FE PC `views/be/{그룹}/{메뉴}`, FE 모바일 `views/bm/{그룹}m/{메뉴}m`.

## 운영 방침 (2026-06-19 결정)

이 색인은 **메뉴→경로 표를 박제하지 않는다.** 그 정보는 이미 ① 위 도출 규칙, ② `spec/{프로젝트}/_knowledge/menu-list.md`(메뉴 레지스트리), ③ 형제 BE/FE 레포 실시간 스캔으로 충분히 커버되며, 정적 표로 만들면 소스 변경 시 곧바로 드리프트(실제와 어긋남)가 발생하기 때문이다.

목적별 처리:
- **(a) 메뉴코드 → 코드 위치 찾기** → 별도 색인 불필요. 위 도출 규칙으로 바로 계산하거나, 필요 시 BE/FE 레포를 직접 스캔(`glob`)한다.
- **(b) 존재 여부 / 설계↔소스 드리프트 파악** → 정적 색인이 아니라 **검증 스킬 `/KB_200`** 으로 그때그때 확인한다.
- **(c) 규칙으로 도출되지 않는 것**(공통 모듈·유틸·인터페이스(SIF) 위치, 네이밍 예외) → **여기에 짧은 수동 노트로** 적는다. 이것이 이 폴더의 유일한 고유 가치다.

따라서 자동 생성기(`gen-src-index.py`)·전용 스킬은 두지 않는다. (a)/(b)는 도구가 이미 있고, (c)만 사람이 관리한다.
