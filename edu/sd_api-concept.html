1

스킬명

SD_api (API 명세서 자동 작성)

실행: /SD_api {메뉴코드}

2

주요 작업 내용

화면설계·DB 설계·BE/FE 소스를 분석하여 Interface ID · HTTP 메서드 · URL · Request/Response · Bean 설계 · 비즈니스 규칙이 담긴 API 명세서(-05-api.md)를 작성한다.

3

스킬 개요

DB 설계(/SD_db) 완료 후 실행하는 스킬이다. DB 설계 문서(-03-data-model.md)를 최우선 참조하고, BE Controller · Comp · Bean · Vue 파일을 읽어 화면에서 실제 호출하는 API 전체를 명세한다.

BE 소스가 아직 없는 신규 기능은 ui.md · wireframe · DB 설계로부터 API를 도출해 명세를 먼저 작성할 수 있다. 산출물 -05-api.md는 이후 /PI_be_all로 BE 개발을 시작할 때 핵심 입력 문서가 된다.

4

육하원칙

누가개발 담당자가 Claude와 함께 (AI가 소스 분석·명세 작성 수행)
언제DB 설계 문서(-03-data-model.md)가 준비된 뒤 (/SD_db 완료 후)
어디서AI 허브(common-system-ai) CWD에서 /SD_api {메뉴코드}
무엇을BE Controller·Comp·Bean·FE Vue의 API 호출 패턴과 DB 설계를 종합하여
어떻게분석 범위(BE 포함/FE만/BE 기준)를 사용자와 합의 후 표준 템플릿으로
왜BE 개발(/PI_be_all) 착수 전 API 계약을 문서화하기 위해

실행 순서 (5 Steps)

  • Step 0
    레포 경로 결정 (BLOCKING) AI_DIR / WS / PROJECT / BE_DIR / FE_DIR 변수 도출 — repo-paths.md 규칙 적용
  • Step 1
    기능 정보 파악 메뉴코드 · 기능명 · 도메인(MDM/IW/OW/IV/RT/SIF) 확인
  • Step 2
    기존 파일 확인 → 분석 범위 확정
    케이스 Aapi.md 이미 존재 → 파일 읽어 내용 업데이트
    케이스 Bapi.md 없음 + BE 소스 있음 → 분석 범위 질문 (BE+Vue / FE만 / BE만)
    케이스 C신규 기능 → 폴더 생성 후 ui.md·DB설계 기반으로 신규 작성
  • Step 3
    관련 문서 읽기 (BLOCKING) -03-data-model.md → -02-ui.md → wireframe → DB 스키마 → BE Controller/Comp/Bean → Vue → API 네이밍 규칙
  • Step 4
    api.md 작성 기본정보 → 기능개요 → API 목록 → API 상세 → 사용 테이블 → Bean 설계 → 비즈니스 규칙 → 유효성 검증
  • Step 5
    완료 안내 산출물 위치 확인 → 다음 단계 /PI_be_all 또는 /PI_be_mapper → /PI_be_dao → /PI_be_comp 안내
