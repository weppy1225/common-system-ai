# Git 작업 규칙 (3개 레포 공통)

workspace 하위 3개 레포는 git 운영 방식이 다르다. 작업 전 **어느 레포에서 작업 중인지** 먼저 확인한다.

| 레포 | 운영 개념 | 방식 |
|---|---|---|
| AI 허브 (현재) | 없음 | main 직접 푸시 |
| BE 레포 | 있음 | 운영 전/후로 다름 |
| FE 레포 | 있음 | 운영 전/후로 다름 |

---

## 1. AI 허브 (현재 디렉토리)

- **운영 개념 없음.** `main` 에 **직접 커밋·푸시**한다. 브랜치·PR 불필요.
- 단, **대량 작성·대규모 변경**(다수 파일 생성/이동, 구조 변경 등)이 발생하면 **사용자에게 먼저 묻고** 진행한다.

## 2. BE/FE 레포

운영 시점 = **운영서버가 가동되어 고객이 사용하기 시작하는 시점**.
**현재 운영 전/후인지는 클로드가 매번 사용자에게 확인한다.** (추정 금지)

### 2-1. 운영 전 (개발·테스트 단계)
- 소스가 수시로 바뀌므로 빠른 반영 우선.
- **`test-deploy` 브랜치(테스트 서버 배포용)에 직접 푸시.** 브랜치 안 따고 **PR 없음.**

### 2-2. 운영 후 (고객 사용 시작 이후)
- **레드마인 이슈 번호로 브랜치를 딴다.** 브랜치명 형식: **`#16781`** (`#` + 이슈번호).
- 작업 후 **PR 생성(요청)은 작업자가** 한다.
- **PR 머지는 PR 관리자(`weppy1225` 등 특정 계정)만** 한다.
- 따라서 **클로드는 커밋·푸시·PR 생성까지만** 하고, **`gh pr merge` 로 머지하지 않는다.** (사용자가 명시 요청 시에만 예외)

## 3. 신원 구조 (이 머신 기준)

커밋 신원(`git config user.name`)과 GitHub 작업 신원(`gh` 인증 계정)은 **별개**다. 또한 Claude 로그인 사용자(실제 작업 지시자)와도 다를 수 있다.

> 신원 3종(Claude 로그인·git commit·gh)의 의미·도출법·주의점(gh 계정이 PR 관리자면 머지 금지 등) → **`SET_USER_USAGE.md`**. 이 머신의 실제 값은 `SET_USER.md`(`.gitignore` 대상, 커밋 금지). 값은 머신·작업자마다 다르므로 여기 하드코딩하지 않는다.
>
> ⚠️ 이 머신의 gh 인증이 **PR 관리자 계정**일 수 있다. 그 경우 `gh pr create` 를 돌리면 관리자 이름으로 PR 이 열린다 — 규칙상 "PR 생성은 작업자"이므로, 작업자 신원으로 PR 을 열어야 하면 gh 재로그인이 필요하다. (현재 머신 값은 `SET_USER.md` 참조)

## 4. 공통 규칙

- 커밋 메시지는 **한글**로 작성한다.
- 클로드가 만든 커밋은 본문 끝에 `Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>` 트레일러를 남긴다.
- 커밋·푸시·PR 생성은 **사용자가 명시 요청할 때만** 한다.
- main(또는 test-deploy) 직접 푸시 전 `git pull --ff-only` 로 최신 여부를 확인한다.
