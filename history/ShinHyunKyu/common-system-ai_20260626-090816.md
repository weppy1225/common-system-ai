# 작업 히스토리

## DB↔앱 네이밍 케이스(snake_case↔camelCase) 4곳 중복 정리 → naming-rule §2.2.1 SoT 위임 (SoT 중복 #9 해소)

### 변경 내용
1. **patterns/20-database/20-rule/01-naming-rule.md §2.2.1 — 레이어별 케이스 규칙 정본 블록 신설(SoT 보강)**
   - 이전: 컬럼 = `lower_snake_case` (DB쪽만) 정의. 앱/Java/FE camelCase·변환 지점은 없음.
   - 이후: "레이어별 케이스 규칙 (SoT)" 소절 추가 — DB(snake_case)↔앱(camelCase) 표 + 변환 지점(MyBatis `<resultMap>`/`AS camelCase`) + 서브쿼리 별칭 snake_case 유지. "정본, 타 문서는 참조하고 중복 금지" 명시.
   - 사유: 케이스 규칙이 4곳에 단면별로 흩어져 SoT 부재. 네이밍 정본 문서에 전체 규칙을 모아 단일 SoT 확립.

2. **patterns/40-frontend/20-convention/01-naming.md §5 — 한 줄 순수 중복 → 포인터**
   - 이전: `규칙: DB는 snake_case, FE 는 camelCase`.
   - 이후: 케이스 규칙 한 줄 + `정의·변환 지점 SoT → 20-database/20-rule/01-naming-rule.md §2.2.1` 포인터.

3. **patterns/40-frontend/10-architecture/02-be-fe-contract.md §3 — 케이스 정의 포인터화, 계약 고유 정보 유지**
   - 이전: 표에 `DB snake_case`·`Java/FE camelCase` 정의 + resultMap 변환 + 동적컬럼 AS.
   - 이후: 표 위에 케이스 정의 SoT 포인터 추가. 표는 BE↔FE 계약 관점만 — `FE는 camelCase 수신`·`resultMap 변환`·`동적컬럼 AS camelCase 명시` 유지(중복 정의 행 제거).

4. **patterns/20-database/30-convention/01-sql-query-style.md §1 — SQL 토큰 케이스표 유지 + SoT 포인터 1줄**
   - 이전: §1 대소문자표(컬럼 소문자·AS 별칭 camelCase 등)만.
   - 이후: 표 직후 포인터 추가 — `DB↔앱 케이스 정의 SoT → 20-rule/01-naming-rule.md §2.2.1, 여기 컬럼소문자·AS camelCase는 그 규칙의 SQL 적용`. 표 자체는 SQL 작성 인라인 참조라 유지.

### 비고
- 4곳이 동일문장 복붙이 아니라 단면별(DB정의/SQL변환/BE↔FE계약/FE한줄)로 분산 → 각자 고유 가치(SQL 토큰표·BE↔FE 계약·동적컬럼 AS)는 보존하고 케이스 정의만 SoT로 위임.
- 검증: `check-doc-refs.py` 깨진 참조 10건(기존 baseline 동일, 증가 없음). 신규 추가 링크(naming-rule §2.2.1·02-mybatis-convention) 모두 정상 해석. PYEXIT=1 은 기존 OMS spec-상대경로 오판 건으로 이번 변경 무관.
