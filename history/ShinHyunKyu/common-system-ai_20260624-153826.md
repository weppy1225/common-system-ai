# 작업 히스토리

## 30-backend 문서 3계층(Rule/Convention/Guide) 정의 + API URL 패턴 중복 제거

### 변경 내용
1. **`patterns/30-backend/00-overview.md` — 3계층 정의 정식화**
   - 이전: 본문 "현재 정의 중" stub
   - 이후: Rule(강제)/Convention(관례)/Guide(권장) 정의 표 + 이 레포 `20-rule/` 보충 설명(언어강제 아니나 프로젝트 계약상 강제 취급) + 중복 금지(SoT) 원칙·주제별 SoT 표·하위 구조
   - 사유: rule/convention/guide 경계가 미정의(stub)라 같은 주제가 여러 폴더에 중복 박제되는 원인이 됨

2. **`patterns/30-backend/30-convention/01-coding-convention.md` §4 — 중복 표 제거**
   - 이전: Controller URL 패턴 코드블록 표를 자체 보유(SoT인 api-naming-rule과 드리프트: `{메뉴코드_인스턴스}` 표기·`excel/valid` 행 등 불일치)
   - 이후: SoT(`20-rule/01-api-naming-rule.md §2`) 포인터 한 줄로 교체
   - 사유: API URL 패턴은 rule 영역 SoT 단일화. 중복 박제로 인한 드리프트 제거

3. **`patterns/30-backend/20-rule/01-api-naming-rule.md` §2 — 누락 행 이관(소실 방지)**
   - 이전: 엑셀 업로드 유효성 검사 API 행 없음
   - 이후: `{메뉴코드}_POST_{리소스}_EXCEL_VALID | POST | .../excel/valid | 엑셀 업로드 유효성 검사` 행 추가
   - 사유: coding-convention §4에만 있던 `excel/valid` 행을 삭제 전 SoT로 이관해 정보 소실 방지
