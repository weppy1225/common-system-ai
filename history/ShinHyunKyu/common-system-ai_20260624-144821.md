# 작업 히스토리

## API 네이밍 규칙(01)에 새어든 메뉴코드 내용 제거 — 02와 역할 분리

`01-api-naming-rule.md`(API 규칙)와 `02-menu-code-rule.md`(메뉴코드 규칙)는 목적이 다른데, `01`이 메뉴그룹표(§2)·메뉴코드 생성규칙(§3)을 추가로 담고 있어 `02 §1`(메뉴코드 규칙·업무약어표)과 중복되고 있었다. 두 파일을 목적대로 분리(01=API, 02=메뉴코드)했다.

### 변경 내용
1. **01-api-naming-rule.md 슬림화** (`patterns/30-backend/20-rule/01-api-naming-rule.md`)
   - 삭제: §2 메뉴그룹표(IW1000…12개), §3 메뉴코드 생성규칙(그룹2+업무2+유형2 → MDPD01)
   - 대체: "메뉴코드 체계·그룹·약어·생성규칙 SoT는 02-menu-code-rule.md" 포인터 1줄
   - 섹션 재번호: §4 API 생성규칙→§2, §5 업무절차→§3
   - frontmatter: description에서 "메뉴코드 체계" 제거, tags에서 `menu-code` 제거, version 1.0.0→1.1.0
   - 결과: 01은 시스템구분(§1) + API 생성규칙(§2) + 업무절차(§3) 의 순수 API 규칙으로 정리

2. **02-menu-code-rule.md 보강** (정보 보존 + 라우팅)
   - §1: 01 §3에 있던 코드 분해 설명(4자리약어 = 그룹2 + 업무영단어2, MDPD01/CT 예시) 1줄 추가 → 01에서 들어낸 내용 보존
   - §2: "HTTP 메서드별 API 인터페이스 ID·전체 URL 명세는 01-api-naming-rule.md 참조" 라우팅 1줄 추가

### 결과
- 중복(01 §2·§3 ↔ 02 §1) 제거. 01=API 네이밍, 02=메뉴코드 체계로 목적 완전 분리.
- 내용 손실 없음: 메뉴그룹/약어표는 02가 상위집합(18개), 코드 분해 예시만 02로 이전.
- 두 patterns 파일은 다른 문서에서 상대링크 참조되지 않아 참조 갱신 불필요(확인 완료).

### 비고
- "API URL 패턴 4중 정의(01 §2 ↔ 02 §2 ↔ coding-convention §4 ↔ be-fe-contract §1)"는 이 둘 외 2개 파일까지 걸쳐 별도 정리 대상으로 남김.
