# 작업 히스토리

## API 네이밍 SoT(api-naming-rule §2) 실측 정정 — 등록 PUT/수정 PATCH + 복수건 Save 규약

### 배경 (근거: BE/FE 실제 코드 전수 조사)
- BE Controller 86개 조사 결과 등록=`@PutMapping`(18)·수정=`@PatchMapping`(26)이 지배적인데, SoT 문서는 소수 패턴(`POST .../insert`·`/update`, 9건)만 기재 → stale.
- `POST .../insert`·`/update` 9건은 대부분 **파일첨부(multipart)** 케이스(MDPD01·MDLP01·SMBD01). MDCP01만 파일 없는 레거시 예외.
- 복수건 일괄은 `{메뉴}Save{리소스}` Bean(`insertList`/`updateList`/`deleteList`) 24개 메뉴에서 사용 — 문서의 `rowStatus` C/U/D 방식과 다름.

### 변경 내용
1. **`patterns/30-backend/20-rule/01-api-naming-rule.md` §2 전면 정정**
   - 이전: 등록=`POST .../insert`, 수정=`POST .../update` 단일 표 + `rowStatus` 기반 save 설명
   - 이후:
     - §2.1 메서드 분기 기준 신설 (단건 JSON=PUT/PATCH, 파일=POST insert/update, 복수건=POST save, 엑셀=PUT excel, 삭제=DELETE)
     - §2.2 인터페이스 ID·URL 표에 `_PUT_{리소스}`(등록)·`_PATCH_{리소스}`(수정) 행 추가, 기존 POST insert/update는 "파일첨부 시"로 단서
     - §2.3 복수건 일괄 저장 규약을 `{메뉴}Save{리소스}` Bean + 3개 분리 List(`insertList`/`updateList`/`deleteList`)로 정정 (rowStatus 드리프트 수정)
     - 레거시 예외(MDCP01) 각주 처리
   - 프런트매터 version 1.1.0→1.2.0, author/last_modified_by 추가
   - 사유: SoT 문서가 실제 코드와 어긋나 있어 신규 개발 오작성 위험

2. **`patterns/40-frontend/10-architecture/02-be-fe-contract.md` §1 SoT 포인터 추가**
   - 이후: BE 메서드 분기 정의처(api-naming-rule §2) 상호참조 한 줄 추가. FE 표(등록=PUT·수정=PATCH)가 SoT와 일치함을 명시
   - 사유: 양 문서 일관성 — FE는 이미 정확했으므로 포인터로 연결

### 비고
- be-fe-contract §1은 당초 "중복"으로 봤으나 실측 결과 FE가 맞고 BE SoT가 stale이었음 → SoT를 코드에 맞춰 정정하는 방향으로 처리(FE 문서는 유지).
- BE/FE 형제 레포는 읽기 전용으로 조회만 함(코드 변경 없음).
